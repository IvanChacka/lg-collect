1. 状态设计规范 (State Design) —— 核心中的核心

状态（State）是 LangGraph 的灵魂。整个流转过程就是对 State 的不断修改。

强类型优先：永远不要使用普通的 dict，必须使用带有类型提示的 TypedDict 或 Pydantic BaseModel。这能让 IDE 给你完美的自动补全，避免拼写错误。

按需更新 (Reducers)：默认情况下，节点返回的数据会覆盖旧 State 里的同名字段。如果你需要“累加”数据（比如把多篇文章塞进一个列表，而不是覆盖上一篇），必须使用 Annotated 和特定的 Reducer 函数（如 operator.add）。

避免 State 臃肿：不要把整个大语言模型的完整响应（包含各种 tokens 统计信息的巨型 JSON）全都塞进 State。只提取你需要的信息（如 selected_topic, final_script）放进去。

2. 节点开发规范 (Node Functions)

节点应该像工厂里的标准化处理单元。

“纯函数”思维：节点函数应该尽可能简单，只负责“读取状态 -> 调用外部逻辑 -> 返回要更新的状态”。

逻辑解耦：把复杂的业务逻辑（比如“调用大模型”、“爬取网页”、“发送飞书 API”）抽离成普通的 Python 工具函数。节点里只写调用这些工具函数的代码。

反例：在节点函数里写了 200 行请求抖音 API、解析 JSON、处理异常的代码。

正例：在单独的 utils/scraper.py 里写爬虫，在节点里只需一行 data = scrape_douyin_hot()。

节点独立可测试：在把节点放入 Graph 之前，你应该能通过手动构造一个模拟的 State 字典，单独调用这个节点函数并测试其输出。

3. 控制流与路由规范 (Edges & Routing)

不要写出复杂的“死循环”或难以追踪的路径。

使用条件边 (Conditional Edges) 处理判断：比如，AI 发现今天没有合适的科技/医疗热点，工作流应该直接结束，而不是继续走后面的发飞书流程。这就需要用到条件边。

明确路由返回类型：你的条件路由函数（Routing Function）应该返回下一个要执行的节点的字符串名称，清晰明了。

4. 动态配置规范 (Configuration)

你的工作流在不同的环境下（测试、生产）可能需要不同的参数。

使用 RunnableConfig 传参：不要把模型名称（如 gpt-4o 或 gemini-pro）、飞书的 webhook 地址、API 密钥硬编码（写死）在节点里。使用 LangGraph 内置的 RunnableConfig 在运行 invoke() 时动态传入。

5. 中断与持久化规范 (Checkpointing & Human-in-the-loop)

这是你这个项目最依赖的特性。

明确 Thread ID 管理：每次触发运行（比如每天早上 9 点），必须为其分配一个全局唯一的 thread_id（比如 hot_topic_2026_04_21）。飞书回调时，必须带着这个 ID 才能唤醒对应的任务。
